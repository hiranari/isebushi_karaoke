# ドキュメント作成ガイドライン

## 目的

プロジェクトの一貫性のあるドキュメント作成を確保し、ドキュメントの保守性と可読性を向上させる。

## ディレクトリ構造

```
docs/
├── architecture/      # アーキテクチャ設計文書
├── compliance/       # コンプライアンス関連文書
├── development/      # 開発関連文書
│   ├── debug/       # デバッグ関連
│   ├── guides/      # 実装ガイド
│   ├── issues/      # 課題管理
│   └── tasks/       # タスク管理
├── guidelines/       # 各種ガイドライン
│   ├── application/ # アプリケーション固有のガイドライン
│   └── development/ # 開発プロセスのガイドライン
└── README.md        # プロジェクト概要
```

## ファイル命名規則

- すべてのドキュメントファイルは **ケバブケース** を使用
  ```
  ✅ audio-quality.md
  ❌ AudioQuality.md
  ❌ audio_quality.md
  ```

- 拡張子は `.md` （Markdown）を使用
- フォルダ名は小文字のみを使用

## Markdownの書式ルール

### 1. 見出しレベル
- H1（#）: ドキュメントのタイトルのみ
- H2（##）: メインセクション
- H3（###）: サブセクション
- H4以降: 必要に応じて使用

### 2. コードブロック
- 言語指定を必ず含める
  ```dart
  void main() {
    print('Hello');
  }
  ```

### 3. リスト
- 箇条書き（-）を使用
- ネストは2レベルまで
- 順序付きリストは必要な場合のみ使用

### 4. 表
- ヘッダー行を必ず含める
- 列の配置を統一（左揃えを推奨）

### 5. 強調
- **太字**: 重要な用語や注意点
- *イタリック*: 補足情報
- `コードスパン`: 変数名、メソッド名、ファイル名

## ドキュメントの種類と必須セクション

### 1. アーキテクチャドキュメント
- 概要
- システム構成
- コンポーネント説明
- データフロー
- 依存関係

### 2. API ドキュメント
- エンドポイント
- パラメータ
- レスポンス形式
- エラーハンドリング
- 使用例

### 3. 実装ガイド
- 目的
- 前提条件
- 実装手順
- コード例
- テスト方法

### 4. トラブルシューティングガイド
- 問題の症状
- 考えられる原因
- 解決手順
- 予防策

## レビュープロセス

### ドキュメントレビューチェックリスト
- [ ] 正しいディレクトリに配置されているか
- [ ] ファイル名が命名規則に従っているか
- [ ] Markdown書式が適切か
- [ ] 必須セクションが含まれているか
- [ ] 誤字脱字がないか
- [ ] コード例が最新か
- [ ] リンクが正常に機能するか

## 保守とアップデート

### 更新のタイミング
- 機能追加・変更時
- バグ修正時
- 仕様変更時
- 定期的なレビュー時（四半期ごと）

### 変更履歴の記録
```markdown
## 変更履歴

| 日付 | 変更内容 | 担当者 |
|------|----------|--------|
| YYYY-MM-DD | 変更の簡潔な説明 | 名前 |
```

## テンプレート

### 新規ドキュメントテンプレート
```markdown
# [ドキュメントタイトル]

## 概要
[このドキュメントの目的と範囲を簡潔に説明]

## 前提条件
- 必要な環境
- 必要な知識
- 必要なツール

## 本文
[メインコンテンツ]

## 参考リンク
- [関連ドキュメント]
- [外部リソース]

## 変更履歴
| 日付 | 変更内容 | 担当者 |
|------|----------|--------|
| YYYY-MM-DD | 初版作成 | 作成者名 |
```

## 注意事項

1. **最新性の維持**
   - 定期的なレビューと更新
   - 古いドキュメントの明確な表示または削除

2. **一貫性の確保**
   - 用語の統一
   - フォーマットの統一
   - スタイルの統一

3. **アクセシビリティ**
   - 画像には代替テキストを提供
   - 色のみに依存しない説明
   - 簡潔で明確な言葉使い

4. **セキュリティ**
   - 機密情報の排除
   - 認証情報の非掲載
   - 内部URLの適切な扱い

## まとめ

このガイドラインに従うことで、以下が実現できます：
- 一貫性のある文書作成
- 効率的な情報共有
- ドキュメントの保守性向上
- チーム間のコミュニケーション改善
